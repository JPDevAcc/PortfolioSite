Description

Summary

This was the final project undertaken with the Developer Academy, and ran for four weeks. Like previous projects, it made use of the MERN stack.

Groups could select a website theme of their choosing, with some optional suggestions provided by the instructor. Our group of three settled on a fitness-app with features such as work-out and dietary recommendations along with social posts and messaging.

There was a strong emphasis on project-management and agile methodologies, and to that effect we made use of tools such as user-stories, weekly sprints, and daily-standups, all of which helped us to plan and prioritise features for implementation.

Division of work

We generally allocated specific features / tasks to individual team members. This had the advantage of allowing us to work mostly independently and helped to avoid complicated merge-conflicts. It also meant that everyone had a roughly equal opportunity to gain further familiarity with both front-end and back-end development.

My main contributions
  • User registration / Account management options
  • Site preferences / Profile settings
  • Dashboard (with notifications, goal progress, and BMI calculator)
  • Contacts / messaging system
I also assisted with the back-end authorization and session-management code and was the manager for the two GitHub repositories.

The development process

This started with the creation of around fifteen user-stories from hypothetical user-groups who we believed would be interested in using our app. Following from this, we created a related list of development tasks, with priority given to those requirements which we felt were more critical to the success of the site, some of which were then allocated to the first sprint.

The remainder of this first week was then spent developing a basic website framework with working authentication and session-handling. We decided to use express-session in conjuction with memorystore for the latter, which worked well for us after some initial cookie-related difficulties. We also experimented with an external API to retrieve themed images from the Unplash image-library.

The second week saw the addition of account-management options, and a profile "goals" tab. Sections were also added which drew on data from external APIs.
These included an "Exercises" section, drawing data from ExerciseDB, and a "Recipes" section using data from Spoonacular (Recipe - Food - Nutrition API).

The third week saw the implementation of social posts, the contacts / messaging / notifications system, a BMI calculator tool, and a section where users could attempt various challenges such as doing a certain number of exercises or activities within a specified time-span.

In the final week, there was a greater emphasis on testing, bugfixing, and validation / security, but we also had the opportunity to add a few more features and these included a "custom workouts" section and a weight-goal progress bar (with weight history recorded by the back-end). A further minor addition was the inclusion of a "loading spinner", displayed when network requests exceeded a certain time threshold.

Retrospective

This was a challenging project. We were able to fully or partially satisfy most of the 'core' requirements but some did prove too ambitious, given the project time-span, and our limited familiarity with some of the technologies in use. Nevertheless, the team worked well together, and the functionality we did implement worked well for the most part, with just a few known issues remaining at the end of the four-week period.

I personally found it to be a valuable learning experience, especially in the opportunity it afforded to gain a better understanding of MongoDB (and non-relational databases in general), and also to further improve my GitHub management skills.

In terms of possible improvements if such a project were to be carried out in the future, I believe that tasks could be prioritised a little more effectively, as there were a couple of days spent on features that were abandoned due to technical problems while other more critical development tasks were still outstanding.

It's also possible that the project would have benefitted from a test-driven-development approach, but given the additional time required to write tests covering all functionality, it's not certain whether this would have led to a better outcome.

Finally, in terms of improvements that could be made to the site as it currently stands, there are a number of areas / features that could be prioritised. These include fixing any remaining known issues and extending automated test coverage, improved front-end and back-end validation, better account security (with e-mail address validation and optional 2FA), and support for finding users by approximate location.
